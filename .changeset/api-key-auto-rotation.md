Per-key auto-rotation for API-key authentication. When an upstream LLM provider starts returning 401/403/408/409/429/5xx for a specific API key, Manifest now records the failure, computes an exponential cooldown (60s → 120s → 240s … capped at 1h) on that key, and skips it from `selectProviderKey` until the cooldown expires. A successful forward immediately resets the streak and clears the cooldown. The Connect modal surfaces the cooldown as a "cools down in 4m 12s" badge with a warning icon, so users can see when their keys are exhausted without grepping the database. `MAX_KEYS_PER_PROVIDER` is raised from 5 to 20 to cover providers that cap usage on a per-key rolling window (e.g. AeroLink's $10/4h cap).

Schemas: adds `last_failure_at`, `consecutive_failures`, and `cooldown_until` columns to `tenant_providers` (migration `AddTenantProviderKeyHealth1793000000000`, idempotent on re-run). Backwards compatible — existing rows default to `consecutive_failures = 0`, `cooldown_until = NULL`, so they're immediately eligible. Cooldown updates invalidate the tenant-scoped provider-key cache so the next request sees the new `cooldown_until` instead of the 2-minute-stale snapshot.

AeroLink model prices now flow from the docs-pinned catalog into the live `/v1/models` response: the parser fills in `inputPricePerToken` / `outputPricePerToken` per model id from the existing `AEROLINK_PRICES` map (5/1M input for `claude-opus-4-6`, 3/1M for `claude-sonnet-4-6`, etc.), so the Routing → available-models list and the per-agent provider tile show real prices instead of `—`. Models the docs don't cover yet keep `null` prices and surface as "Price not set" in the UI — the alternative (overwriting with $0.00) was worse because it would silently undercharge.

Connect modal: pin `isConnectedApiKey` / `isSubscriptionConnected` lookups to the active row only, so a multi-key chain where the first DB row happens to be `is_active=false` (e.g. a recently-disabled key) no longer masks the active key and flips the modal back to the first-time connect form. The list view continues to enumerate `activeKeys()` (active + has_api_key) as before.

In-request key rotation: when a request hits a 401/403/408/409/429/5xx on the first key in the chain, the proxy now swaps to the next eligible key in the **same request** (driven by `selectNextEligibleKey`) instead of failing the request and waiting for the cross-request cooldown to expire. The cross-request cooldown is still what gates the *next* request, so the two stay aligned — a key that's hot right now is also the one the next request skips. Model-level errors (400, 404, 422) don't trigger rotation since a different key would fail the same way. Pinned keys (structured route `keyLabel`, explicit `providerKeyLabel`) are honored — silently swapping a pinned key would mask routing-pin misconfigurations as successful retries.
